3 · Logic & Edge-Case Notes (for dev hand-off)

Prize pool: 500 total prize slots, distributed across tiers. Each tier has its own counter (e.g., 25% off = 120 slots, 50% off = 30 slots). When the counter for a given tier hits 0, the wheel still shows the segment but the spin resolver re-rolls to "Better luck next time" (Popup B) — this prevents the user from seeing an empty wheel while keeping the UI consistent.

Email validation: Screen 1 hits the Imagine Art user DB. Only exact-match registered emails pass. Case-insensitive, trim whitespace. Rate-limit to 5 attempts per IP per hour to prevent email enumeration.

One spin per user: Track by email + device fingerprint. If the same email returns, show a "You've already spun — here's your code again" state (this is a future addition, not in the current flow).

Screenshot & DM mechanic: Because redemption is manual via social DM, the prize code on Screen 3 must be unique per user (e.g., SPIN-25-X7K9) so admin can invalidate after use. Store the code server-side with status: issued → claimed.

Expiry: Codes valid 48 hours from issue. Admin response SLA: 24h.

Analytics to capture: link taps by source (IG vs X), email submit rate, email-not-found rate, spin completion, prize distribution, screenshot-confirm tap rate, DM-to-admin conversion.

Anti-abuse: Hide the prize-allocation logic server-side. Never expose "slots remaining" in client responses. Verify the email → spin → code issuance chain server-to-server.

Screens NOT yet designed (future v2): Already-spun state · Expired-code state · Admin dashboard for verifying screenshots.
